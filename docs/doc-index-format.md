# Structured Documentation Index

A JSON format for per-symbol package documentation, designed so GUI
applications can show rich help without compromising Maxima's CLI help
system.

## Problem

Maxima's CLI needs `.info` files and `-index.lisp` for `?` and `??`.
GUI applications (VS Code extension, web UIs, notebook frontends) need
something richer: per-symbol documentation with images, examples, and
cross-references.

Today `mxpm doc build` generates `.info` + `-index.lisp` (and optionally
mdBook HTML). But there is no structured, per-symbol format a GUI can
query for inline help, hover docs, or a help panel.

Packages vary in what they ship:

| Package type | Source format | What exists today |
|---|---|---|
| New packages (mxpm scaffolded) | `.md` with heading conventions | `.info`, `-index.lisp`, mdBook HTML |
| Legacy packages | `.texi` or nothing | `.info` + `-index.lisp` if lucky |
| Maxima built-ins | `.texi` | `.info`, `-index.lisp`, HTML manual |

A GUI app shouldn't need to handle all these cases differently.

## Solution: `<pkg>-doc-index.json`

A single JSON file generated by `mxpm doc build` that contains every
documented symbol with its markdown documentation. GUI apps consume this
one file and render markdown themselves; CLI help continues to use
`.info` + `-index.lisp` as before.

### Generation sources

The doc index can be generated from either source format:

1. **Markdown** (preferred for new packages) ŌĆö parse the heading
   conventions (`### Function: name (args)`), extract per-symbol
   sections.
2. **Texinfo / `.info`** ŌĆö parse `@deffn`/`@defvr` blocks, extract text.
   Less rich (no images) but covers legacy packages.
3. **No docs** ŌĆö empty index. GUI shows "no documentation available".

### File layout after `mxpm doc build`

```
my-package/
  doc/
    my-package.md              # author's source (markdown)
    my-package.info            # for Maxima ? and ??
    my-package-index.lisp      # for Maxima help lookup
    my-package-doc-index.json  # for GUI apps  ŌåÉ new
    book/                      # mdBook HTML (optional)
```

## Format

```json
{
  "version": 1,
  "package": "my-package",
  "source": "doc/my-package.md",
  "symbols": {
    "my_solve": {
      "type": "Function",
      "signature": "my_solve(expr, vars)",
      "summary": "Solve expr for vars using the foo method.",
      "body_md": "Full markdown body including examples...",
      "examples": [
        {
          "input": "my_solve(x^2 - 1, x);",
          "output": "[x = -1, x = 1]",
          "description": "Solve a simple quadratic"
        }
      ],
      "see_also": ["my_solve_opts", "my_verbose"]
    },
    "my_verbose": {
      "type": "Variable",
      "signature": "my_verbose",
      "summary": "When true, print extra diagnostics.",
      "body_md": "...",
      "examples": [],
      "see_also": ["my_solve"]
    }
  },
  "sections": [
    {
      "title": "Introduction",
      "body_md": "..."
    }
  ]
}
```

### Top-level fields

| Field | Type | Description |
|---|---|---|
| `version` | integer | Format version (currently `1`) |
| `package` | string | Package name |
| `source` | string | Path to the doc source file, relative to package root |
| `symbols` | object | Map from symbol name to documentation entry |
| `sections` | array | Non-symbol documentation sections (introduction, tutorials, etc.) |

### Symbol entry fields

| Field | Type | Description |
|---|---|---|
| `type` | string | `"Function"` or `"Variable"` |
| `signature` | string | Full signature: `name(arg1, arg2)` for functions, `name` for variables |
| `summary` | string | First sentence/paragraph, plain text. For hover tooltips and search results |
| `body_md` | string | Full documentation in Markdown |
| `examples` | array | Structured examples extracted from the documentation |
| `see_also` | array | Symbol names that are related. Enables cross-reference navigation |
| `category` | string? | Optional category (e.g. "Calculus", "Algebra"). Used by core docs |

### Example entry fields

| Field | Type | Description |
|---|---|---|
| `input` | string | Maxima expression(s) to evaluate |
| `output` | string | Expected result (if shown in the docs) |
| `description` | string | Optional label for the example |

### Section entry fields

| Field | Type | Description |
|---|---|---|
| `title` | string | Section heading |
| `body_md` | string | Section content in Markdown |

## Images

Package authors include images alongside their markdown source using
standard markdown syntax with relative paths:

```markdown
### Function: plot_widget (expr)

Renders an interactive plot of `expr`.

![Example output](images/plot-widget.png)
```

Images stay as relative paths in `body_md`. GUI consumers resolve image
paths relative to the doc-index JSON file location. This keeps the JSON
file small and avoids data URI encoding overhead.

The `.info` path is unaffected ŌĆö texinfo/info is text-only, so images
simply don't appear in CLI `?` help.

## Examples

### Extracting examples from markdown

The existing heading conventions don't define a formal example format.
We adopt a lightweight convention: fenced code blocks with `maxima`
language tag inside a symbol's section are treated as examples.

````markdown
### Function: my_solve (expr, vars)

Solves `expr` for `vars`.

```maxima
(%i1) my_solve(x^2 - 1, x);
(%o1)                      [x = -1, x = 1]
```
````

The pipeline parses these blocks, splitting on `(%i` / `(%o` markers
to separate input from output. Plain fenced blocks without
input/output markers are included as-is with empty `output`.

### Extracting examples from texinfo

Texinfo uses `@example` blocks:

```texinfo
@deffn {Function} my_solve (@var{expr}, @var{vars})
Solves @var{expr} for @var{vars}.

@example
(%i1) my_solve(x^2 - 1, x);
(%o1)                      [x = -1, x = 1]
@end example
@end deffn
```

Same `(%i`/`(%o` parsing applies.

## Cross-references

### Within a package

The `see_also` field lists related symbol names from the same package.
Authors express these in markdown with a convention at the end of a
symbol's section:

```markdown
### Function: my_solve (expr, vars)

...

See also: `my_solve_opts`, `my_verbose`.
```

The pipeline parses the final "See also:" line and populates the
`see_also` array. In texinfo this maps to `@xref` / `@pxref`.

### Across packages

Cross-package references use a `package:symbol` syntax:

```markdown
See also: `diophantine:dio_solve`.
```

This generates `"see_also": ["diophantine:dio_solve"]` ŌĆö the GUI can
resolve these if the referenced package is installed and has a doc
index.

## How GUI apps use this

### Hover documentation

When the user hovers over `my_solve` in the editor:

1. LSP looks up `my_solve` in the doc index.
2. Shows `summary` as the first line, `signature` formatted as code.
3. Optionally shows the first example.

### Help panel

When the user asks for full docs (e.g. F1, `?` command):

1. Render `body_md` as markdown in a webview panel.
2. Resolve image paths relative to the doc-index file location.
3. `see_also` entries become clickable links.
4. Examples get a "Copy" or "Run" button.

### Search

The `summary` and `body_md` fields are searchable. The `type` field
enables filtering ("show only functions" / "show only variables").

### Completions

The `signature` and `summary` fields feed into autocomplete:

```
my_solve(expr, vars)  ŌĆö  Solve expr for vars using the foo method.
```

## Generation pipeline

### From markdown

```
my-package.md
  ŌåÆ parse headings into symbol sections
  ŌåÆ for each symbol section:
      ŌåÆ extract summary (first paragraph)
      ŌåÆ extract examples (```maxima blocks)
      ŌåÆ extract see_also ("See also:" line)
  ŌåÆ non-symbol sections go into `sections` array
  ŌåÆ emit my-package-doc-index.json
```

### From texinfo / .info

```
my-package.info (or .texi ŌåÆ makeinfo ŌåÆ .info)
  ŌåÆ parse @deffn/@defvr nodes (reuse info_index.rs logic)
  ŌåÆ for each definition:
      ŌåÆ extract summary (first paragraph of body)
      ŌåÆ extract examples (@example blocks)
      ŌåÆ extract see_also (@xref references)
      ŌåÆ convert body to markdown
  ŌåÆ emit my-package-doc-index.json
```

### Integration with `mxpm doc build`

The doc index is generated automatically alongside `.info` and
`-index.lisp` ŌĆö no extra flag needed. It becomes a standard artifact
of the doc build pipeline.

## Versioning

The `version` field allows the format to evolve. GUI apps should check
the version and handle unknown versions gracefully (warn and skip, or
attempt best-effort parsing).
