# Package Documentation & Maxima Help System

How Maxima's built-in help (`?` and `??`) works, and what it would take for third-party packages to integrate with it.

## How Maxima's Help System Works

### The Pipeline

Maxima's documentation goes through a multi-stage build:

```
.texi (Texinfo source)
  → makeinfo → .info (GNU Info format)
  → build_index.pl → <topic>-index.lisp (keyword → position lookup)
  → build_index.pl → <topic>-index-html.lisp (HTML cross-references)
```

### Runtime Lookup

When a user types `? foo` or `?? foo`:

1. Maxima loads `*-index.lisp` files at startup, which populate `cl-info::*info-tables*` — an alist mapping topic names to `(filename offset)` pairs.
2. `?` does exact match; `??` does substring match against the topic keys.
3. The matched entry's `(filename offset)` is used to seek into the `.info` file and display the relevant node.

### Where Files Live

In a standard Maxima install:

- `.info` files: `<prefix>/share/info/` (e.g. `maxima.info`, `maxima.info-1`, …)
- `-index.lisp` files: `<prefix>/share/info/` alongside the .info files
- The core documentation source: `doc/info/*.texi` in the Maxima source tree

### The Build Dependency

`build_index.pl` is a Perl script in the Maxima source tree (`doc/info/build_index.pl`). It parses `.info` files to extract node names and byte offsets, then emits Lisp forms. It is not installed as part of a normal Maxima binary distribution — it's a build-time tool.

## Pain Points for Third-Party Packages

1. **`build_index.pl` is not distributed.** Users installing Maxima from a package manager (apt, brew, etc.) don't have it. mxpm would need to either bundle it, reimplement it, or find another way.

2. **`.info` file location is global.** Maxima looks for `.info` files in `<prefix>/share/info/`, which is outside the user directory. Packages installed to `~/.maxima/<name>/` can't simply drop `.info` files there and have them found.

3. **`*info-tables*` is populated at startup.** There's no built-in hook for packages to register additional documentation at `load()` time.

4. **Texinfo toolchain required.** Authors need `makeinfo` (part of GNU Texinfo) to build `.info` from `.texi`. This is an extra dependency most Maxima package authors won't have.

## Possible Approaches for mxpm

### Option A: Lisp-Level Integration

Packages provide pre-built `.info` + `-index.lisp` files. At `load()` time, a small Lisp shim appends entries to `cl-info::*info-tables*` and adds the package's directory to the info search path.

**Pros:** Integrates with existing `?`/`??` commands natively.
**Cons:** Requires package authors to build `.info` files (needs `makeinfo`). Relies on Maxima internals (`cl-info::*info-tables*`) which could change.

### Option B: mxpm Builds Documentation

mxpm itself runs `makeinfo` + index generation at install time. The package only ships `.texi` source.

**Pros:** Package authors write Texinfo, mxpm handles the rest.
**Cons:** Requires `makeinfo` on the user's machine. mxpm needs to either bundle or reimplement `build_index.pl`. Still needs the Lisp shim for runtime registration.

### Option C: Separate `mxpm doc` Command

Skip native `?`/`??` integration. Packages include structured documentation (Texinfo, Markdown, or plain text) and mxpm provides its own `mxpm doc <package> [topic]` command to search and display it.

**Pros:** No Maxima internals dependency. Works with any doc format. Easy for package authors.
**Cons:** Users must leave the Maxima REPL (or use `system()`) to access docs. Two separate help systems.

### Option D: Do Nothing (For Now)

Packages include a README and in-source comments. Users read docs on GitHub or via `mxpm info`. Defer help system integration until there's demand.

**Pros:** No complexity. Ship features that matter more first.
**Cons:** No discoverability from within Maxima.

## Recommendation

Option D for the near term — focus on the package ecosystem first. Option A is the most valuable long-term path but requires investigation into whether `cl-info::*info-tables*` manipulation is stable across Maxima versions. A prototype Lisp shim that registers a package's `.info` file at load time would be a good spike to validate feasibility.

If pursued, the Lisp shim could be generated by `mxpm init` as part of the package template, so authors who write Texinfo docs get help integration automatically.
